# harmony-cli

A Dockerized prototype that ingests a California HCD manufactured-home title PDF, drives a **GitHub Copilot CLI agent** (model: `claude-haiku-4.5`) over MCP-based PDF tools to extract title data, presents it for human verification (HITL), then fills three blank HCD transfer forms (`HCD 476.6G`, `HCD 476.6`, `HCD 480.5`) and returns a downloadable ZIP packet — all from a single-container deployment at [http://localhost:3031](http://localhost:3031).

---

## Quickstart

```bash
cp .env.example .env
# Add your COPILOT_GITHUB_TOKEN to .env
docker compose up --build
```

Open [http://localhost:3031](http://localhost:3031). Upload a title PDF → review extracted fields → approve → download the filled-forms ZIP.

## Stack & Justifications

| Layer | Pick | Why |
|---|---|---|
| Container base | `node:22-bookworm-slim` + `tini` PID 1 | Required by Copilot CLI v1+ (Node 22) and zombie-reaping for spawned subprocesses. Debian/glibc, not Alpine — `@github/copilot` and `git` are glibc-built. |
| Frontend | **Next.js 15 App Router**, React 19, Tailwind, shadcn/ui, Lucide icons | Single-process model lets Server Routes spawn Copilot directly. `output: "standalone"` keeps the image lean. |
| Agent runtime | **GitHub Copilot CLI `@github/copilot` ≥1.0.24**, model **`claude-haiku-4.5`** *(only model used)* | Headless via `--output-format=json --no-ask-user`; per-run isolation via `COPILOT_HOME`. Haiku is the cheapest/fastest frontier model that still handles tool-calling reliably — fits the ~$0/run target. |
| MCP tools | `mcp-pdf[forms]` + `pymupdf4llm-mcp` (both stdio, both `uvx`) | One server covers `extract_text` + `extract_form_data` + `fill_form_pdf` + `merge_pdfs`; the other gives Copilot clean Markdown context for reasoning. |
| Orchestration | In-process EventBus + SSE, deterministic state machine | Avoids Redis/Kafka. Sync `bus.publish()` with per-subscriber ring buffer (256, drop-oldest). |
| Persistence | `better-sqlite3` for `runs` table; flat files under `/workspace/runs/<id>/` for artifacts | Zero-config, embedded, deterministic for a one-container demo. |
| Validation | Zod v4 at every HTTP boundary | Catches malformed input before any business logic runs. |
| Observability | JSONL transcript per run + structured logger (pino) | Full replay, easy debugging, zero infra. |
| Tests | Vitest (unit/contract) + Playwright (e2e in container) | Layered test pyramid covering schemas, event bus, config, and end-to-end flow. |

## Scaling Limitation (Assignment §6)

> **Pure model-size scaling does not solve our bottleneck.** Our extraction quality is gated by *PDF tool fidelity* (OCR confidence on the scanned title's decal/serial regions, AcroForm field discovery on HCD's blank PDFs) and by *prompt-token throughput per run*. Doubling the model from Haiku to Opus would burn ~5× the cost and ~3× the latency for marginal field-extraction gains; the actual wins come from upgrading the OCR backend (Tesseract → Mistral OCR / Azure Document Intelligence) and from per-form field-name catalogues that shrink the Copilot context window. In short: at our scale the system is **tool-bound**, not **parameter-bound**, so investing in better MCP tools and tighter prompts beats investing in a larger model.

## How I Used AI Tools

| Tool | What I Used It For |
|---|---|
| **GitHub Copilot (Claude Opus 4)** | Primary coding agent — drove all phase implementation via VS Code agent mode. Architecture reasoning, file scaffolding, surgical edits, test authoring, and iterative debugging were all done through structured prompts. |
| **Claude (direct)** | Research synthesis — produced the `references/research/*.md` reference documents by distilling official docs (MCP spec, Copilot CLI, Next.js App Router, Docker best practices) into actionable, project-scoped briefs. |
| **GitHub Copilot CLI (`claude-haiku-4.5`)** | The in-product agent runtime — this is the LLM that runs *inside* the container to extract title fields and fill HCD forms via MCP tool calls. It is the "inner loop" agent. |

### Prompts authored by hand
- `prompts/extractor.md` — Extraction agent persona, field schema, constraints.
- `prompts/filler.md` — Filler agent persona, field catalogue mapping instructions.
- `prompts/field_catalogue.json` — Discovered AcroForm field names for each HCD form (generated via `scripts/discover-fields.ts`, then hand-verified).
- All phase prompts in `PHASE_PROMPT_TEMPLATE.md` and `TODO_PDF_AGENT.md`.

### What was AI-generated and hand-tuned
- **Scaffolding** (layout, components, API routes): generated by Copilot, then hand-tuned for project conventions (Zod boundaries, EventBus wiring, SSE backpressure).
- **State machine** (`src/lib/runs/stateMachine.ts`): architecture designed by hand, implementation generated and iterated with Copilot.
- **Test suite**: test structure and edge cases designed by hand, boilerplate generated by Copilot.
- **Dockerfile**: multi-stage structure designed from reference docs, implementation generated and hardened iteratively.

## API Routes

| Method | Path | Description |
|---|---|---|
| `POST` | `/api/runs` | Create a new run (multipart: upload title PDF) |
| `GET` | `/api/runs` | List all runs |
| `GET` | `/api/runs/:id/events` | SSE stream of Copilot thought events |
| `POST` | `/api/runs/:id/approve` | HITL approval — resume pipeline after human review |
| `POST` | `/api/runs/:id/cancel` | Cancel/reject a run — transitions to `failed` from any non-terminal state |
| `GET` | `/api/runs/:id/download` | Download the filled-forms ZIP packet |
| `GET` | `/api/runs/:id/transcript` | Download the JSONL transcript for a completed/failed run |

## Human in the Loop (HITL)

After the Copilot extraction agent reads the uploaded title, the system pauses and presents the extracted fields to the user for review. The user sees:
1. A **modal overlay** with all extracted fields (decal number, serial number, owners, etc.)
2. The ability to **approve** or reject the extraction before any forms are filled.

The state machine refuses to advance to the form-filling stage until the human explicitly approves via `POST /api/runs/:id/approve`. This ensures no incorrect data propagates into the final HCD packet.

## What I'd Harden Next

- **Multi-page titles**: Support titles that span multiple PDF pages or have attachments.
- **Real OCR upgrade**: Replace Tesseract with Mistral OCR or Azure Document Intelligence for higher confidence on low-quality scans.
- **Audit log**: Track every extraction, approval, and fill event with timestamps and user identity.
- **Multi-tenant**: Add authentication (GitHub OAuth or similar) and per-org data isolation.
- **Field validation rules**: HCD-specific validation (e.g., decal format `LAA####`, serial format patterns).
- **Retry & error recovery**: Allow users to re-run extraction or re-fill forms without starting over.
- **Integration write-backs**: After HCD submission, update accounting, tenant, and property systems automatically.
